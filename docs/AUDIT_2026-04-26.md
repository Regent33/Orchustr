> ## Accepted Technical Debt
>
> Three audit items are tracked but **not fixed in this implementation
> pass** because the smallest correct fix is a breaking redesign that
> does not belong in the same change as the bug-fix batch. They are
> formally accepted as known limitations:
>
> - **#1 — `*Orchestrator` shim structs.** The ~20 zero-field structs
>   that wrap an underlying impl with one `tracing::info_span!` call
>   each remain in the public surface. Removing them would break every
>   downstream `use ...::LoomOrchestrator`-style import in the
>   bindings. Path forward: deprecation cycle (one minor version with
>   `#[deprecated]` and inline `#[instrument]` on the impl methods,
>   then removal). Tracked for `0.2.0`.
> - **#11 — Executor-side state clone in `or-loom`.** The hot-path
>   clone in `ExecutionGraph::execute_from` (one full `T::clone` per
>   node entry) requires either changing handler signature from
>   `Fn(T)` to `Fn(&T)` or having `T::merge` consume both arguments.
>   Both invalidate every existing `add_node` callsite. Handler-local
>   clones inside built-in topologies have already been removed
>   (batch 5); the remaining executor-side clone stays until the
>   contract decision lands in `0.2.0`.
> - **#23 — Three parallel host-language graph executors.** Python /
>   TypeScript / Dart each ship their own pure-host graph executor.
>   Reducing to one Rust source of truth requires an async-callback
>   FFI bridge in each language (pyo3-asyncio, napi-rs Promise, Dart
>   `dart:ffi` `NativeCallable`). Until that lands, the per-binding
>   pyo3/napi stubs (`PyGraphBuilder`, etc.) at least retain the
>   handlers users register so callers can drive the executor in their
>   host language without losing state.

## Executive Summary

Orchustr is structurally ambitious — a hexagonal-architecture (`domain` /
`application` / `infra`) Rust workspace with 30+ crates, 9 tool crates, and
three language bindings. The bones are good: `or-loom` (graph executor),
`or-core` (retry/budget primitives), and `or-lens` (dashboard) are clean,
testable modules. Mockall is wired into the trait definitions and tracing
spans are consistently applied.

**However, the framework is not yet usable end-to-end for the advertised
"build a production agent in Python/TS/Dart" workflow.** The most severe
findings are concentrated at the boundaries between layers:

1. **The Python FFI is largely a façade.** `PyForgeRegistry`, `PyGraphBuilder`,
   and `PyConduitProvider` accept handlers / providers and silently discard
   them. Nothing the Python user supplies actually executes through the Rust
   runtime. The "binding" exists, but it is structurally inert.
2. **The CLI's flagship command does nothing.** `orchustr run` parses the
   project config and hands it to `DefaultProjectRunner`, which is a no-op.
   `orchustr trace` starts the dashboard and immediately shuts it down.
3. **Pause/Resume loses state.** `LoomError::Paused` only carries a
   checkpoint id. Any state mutation produced by the paused node is dropped
   on the floor.
4. **Builder extension point silently breaks.** `SentinelAgentBuilder::topology`
   accepts any `LoopTopology` impl but only three built-in concrete types are
   wired via `Any` downcasting; user-provided topologies fail with
   `LoomError::UnboundNode` at build time.
5. **Massive duplication across bindings.** Each language reimplements the
   graph executor and conduit logic in pure Python / JS / Dart. The Rust
   "engine" is sidestepped — bindings are mostly using `or-bridge` only for
   prompt rendering, state normalization, and a generic
   `invoke_crate_json(crate, op, payload)` JSON RPC.
6. **Over-applied clean architecture.** Almost every crate exposes a
   zero-field `*Orchestrator` whose only job is to add a tracing span
   around a call. ~20 of these exist. For mid/junior developers, the
   indirection adds friction without adding value.

The **good news**: the `or-loom` graph engine, `or-core` budget/retry math,
the Dart FFI (allocation contract is correctly documented and observed),
and the OpenAI/Anthropic conduits in TypeScript are all well-built. The
fix path is targeted, not architectural.

| 32 | 🟢 | All | None of the bindings export an OpenTelemetry / `or-lens` integration end-to-end; spans created in `or-bridge::invoke_crate_json` exist in Rust but never reach a Python/TS subscriber unless the user separately calls `install_global_subscriber`. Document or auto-install. |

### 1.6 Orchestrator Module — Data Flow, Error Propagation, Deadlock Risks

The "orchestrator" surface is split across `or-sentinel` (single-agent ReAct
/ Plan-Execute / Reflection loops), `or-colony` (multi-agent), and `or-loom`
(graph executor). Findings:

#### Data Flow
- Sentinel passes config via `DynState` keys (#4). Every step reads, the
  graph executes, every step strips them. This works but is fragile — a
  user-defined node can stomp `__sentinel_config`. Internal keys are not
  prefixed namespaces, just leading-underscore strings.
- `or-colony` ([crates/or-colony/src/application/orchestrators.rs:27-59](crates/or-colony/src/application/orchestrators.rs#L27-L59)) runs members **strictly sequentially**: each
  worker sees `transcript.clone()` and `initial_state.clone()` and replies
  in turn. The doc says "coordinate" but the topology is a linear pipeline.
  No actual fan-out, no shared message bus, no concurrency. If that is
  intentional, rename `coordinate` → `cascade`. If it is not, this is a
  significant gap relative to the README claim of "multi-agent".

#### Error Propagation
- 🟡 Errors lose layer information. `SentinelError::Loom(error.to_string())`
  ([crates/or-sentinel/src/infra/implementations/sentinel_agent.rs:69](crates/or-sentinel/src/infra/implementations/sentinel_agent.rs#L69)) reduces a typed `LoomError`
  to a string. Same pattern in [crates/or-cli/src/application/orchestrators.rs:114-115](crates/or-cli/src/application/orchestrators.rs#L114-L115).
  Use `#[from]` in `thiserror` to preserve the chain.
- 🟡 Retry layer (#22) does not classify errors and emits a generic
  `Forge("tool retry exhausted")` after `max_attempts` retries, hiding the
  real cause.

#### Deadlock / Concurrency Risks
- 🟡 `InMemoryPersistenceBackend` uses `tokio::sync::RwLock` ([crates/or-core/src/infra/implementations.rs:8](crates/or-core/src/infra/implementations.rs#L8), [crates/or-core/src/infra/implementations.rs:13-15](crates/or-core/src/infra/implementations.rs#L13-L15)). All
  operations are short-lived, but if a node handler holds a read guard and
  re-enters the persistence backend (e.g. via a checkpoint mid-step), it
  will deadlock. No reentrancy guard exists. Document or switch to a
  channel-based store.
- 🟡 `or-bridge::block_on` (#12) routes through `tokio::task::block_in_place`
  when a runtime is current. Under a `current_thread` runtime this panics,
  not deadlocks — but during the panic the JNI/pyo3 GIL state can leak.
- 🟢 `or-loom::ExecutionGraph` is `Clone` and the handler is `Arc<dyn Fn>`,
  so concurrent execution of the same graph is safe in principle. However,
  no test exercises this; if a user shares one graph across threads and a
  handler captures non-Send state internally, the trait bounds are not
  strict enough to catch it at compile time (handlers are `Send + Sync`,
  but their futures are not required to be `Send`). See [crates/or-loom/src/infra/implementations.rs:11](crates/or-loom/src/infra/implementations.rs#L11).

### 2.2 Integration Tests Across Language Bindings

Add a new `tests/cross-lang/` harness that, for each binding, exercises:

1. **Boot + version**: load native lib, call `version()`, assert non-empty.
2. **Catalog round-trip**: `workspace_catalog_json()` → JSON parse → assert
   each crate has the expected operations.
3. **Prompt render**: render `"Hello {{name}}"` with `{"name": "world"}` →
   expect `"Hello world"`. Assert error path on missing var.
4. **State normalize**: pass a JSON object with mixed types, assert
   stable canonical form.
5. **Tool invocation via `invoke_crate_json`**: stand up a `wiremock`
   server in CI, point `or-tools-search` config at it, assert results.
6. **Forge registry callback** (after #14 fix): register a Python tool,
   have a sentinel agent invoke it, assert the Python function ran.
7. **Memory leak smoke**: 10k iterations of render + free; assert RSS
   delta < 10 MiB.

### 2.4 Edge Cases & Failure Modes

- 🔴 **Pause/resume round-trip**: assert state mutations from the paused
  node survive (regression for #9).
- 🔴 **Custom `LoopTopology`**: a topology not in the `Any` triad must
  either work or fail with a clear message (regression for #19).
- 🔴 **Python tool invocation**: Python handler raises → Rust receives a
  typed error, not a stringified traceback (regression for #14).
- 🟡 **Step limit**: graph with a self-loop hits 1024 cap and returns
  `StepLimitExceeded`, not panics.
- 🟡 **OOM oversize payload**: `parse_decision` rejects > 64 KiB ([crates/or-sentinel/src/infra/adapters/parsing.rs:30-36](crates/or-sentinel/src/infra/adapters/parsing.rs#L30-L36)) — verify.
- 🟡 **NUL byte in string**: any user-supplied string crossing FFI must
  either reject (`InvalidInput`) or sanitize before `CString::new`.
- 🟡 **Concurrent pause / resume on same checkpoint id**: which one wins?
- 🟡 **Shell executor timeout**: child is reaped (regression for #21).
- 🟡 **Vector store query with NaN/Inf in input**: cosine similarity
  returns finite or errors — assert.
- 🟢 **Empty roster** in `or-colony::coordinate` → `EmptyColony` error
  (already covered).
- 🟢 **Duplicate node names** in `GraphBuilder::add_node` → second call
  silently rebinds. Decide intentional vs error.

## Phase 3 — Developer Experience Review

### 3.1 API Clarity & Naming Consistency

- 🟡 **Mixed `*Orchestrator` semantics.** Some orchestrators are stateless
  (`SentinelOrchestrator`, `LoomOrchestrator`) and some hold state
  (`ColonyOrchestrator` owns a `roster`). A user has no way to know in
  advance whether to instantiate one per call or per app. Either make all
  of them stateless façades or rename the stateful ones (`ColonyBuilder` →
  build a `Colony`, then call `coordinate`).
- 🟡 **`or-colony::coordinate` vs the multi-agent claim.** Members run
  strictly sequentially. The README's "multi-agent" wording sets a higher
  expectation. Either rename or implement true fan-out.
- 🟡 **Two `GraphBuilder`s in Python.** `_orchustr.PyGraphBuilder` (stub,
  inert) and `orchustr.GraphBuilder` (works, pure Python). Same name, no
  warning in the inert one. Remove the pyo3 stub or relabel it.
- 🟢 **Snake_case + camelCase aliasing in TS** ([bindings/typescript/src/index.js:99-128](bindings/typescript/src/index.js#L99-L128)). Both `addNode` and
  `add_node` work. This is friendly but doubles the surface area; pick
  one (idiomatic JS = camel) and emit a `console.warn` on the other for a
  release before removing.
- 🟢 **`or-cli` subcommand inconsistency.** `Init`, `Run`, `Lint`, `Trace`
  are top-level; `New { Node | Topology }` is nested. Flatten to
  `new-node` / `new-topology` or nest all of them.

### 3.2 Documentation Gaps

- 🔴 No `CONTRIBUTING.md` or workspace-level architecture doc that maps
  the 30+ crates onto the user's mental model. `docs/` exists but is not
  surfaced from the README.
- 🔴 The Python / TS / Dart READMEs do not state which classes are
  Rust-backed vs language-native. Users assume "Rust" because the package
  ships a `.dll`.
- 🟡 `or-bridge::compile_error!` is the first thing many contributors
  hit (#25); it should be mentioned in the workspace README.
- 🟡 Examples in `examples/` are not referenced from any crate doc-test;
  they may bit-rot silently.
- 🟢 `SentinelAgentBuilder::topology` has a `no_run` doc-test ([crates/or-sentinel/src/builder.rs:17-31](crates/or-sentinel/src/builder.rs#L17-L31)) but
  no compile-checked test confirming the *custom*-topology path works
  (it doesn't — see #19).

### 3.3 Onboarding Friction Points

1. **Hello-world ambiguity**: a developer who runs `pip install orchustr`
   and copies the example will produce code where their `register("my_tool",
   handler)` call silently does nothing. No error, no warning. (#14, #15)
2. **`orchustr run` does not run** (#17). The first command in any tutorial
   appears to succeed and then nothing happens.
3. **Crate count**: 30 crates in the workspace; a reader of `Cargo.toml`
   has to guess which to depend on. Provide a `or-prelude` or facade
   crate (`use orchustr::prelude::*`) that re-exports the 5–10 most-used
   types.
4. **`unsafe(no_mangle)` everywhere** in `or-bridge::dart` is a Rust 2024
   idiom but Rust learners on stable channel will not recognise it. Add a
   one-liner in the bridge crate's README.
5. **Trace command misleads**: `orchustr trace` "succeeds" and returns the
   port number while having torn the server down. New users debugging
   "why is the dashboard not at :8080" will have a bad time. (#18)

### 3.4 Three Suggested Simplifications

#### Simplification 1 — Drop the boilerplate `*Orchestrator` shim crates expose

**Before** (current — repeated in ~20 crates):

```rust
// crates/or-loom/src/application/orchestrators.rs
pub struct LoomOrchestrator;

impl LoomOrchestrator {
    pub async fn execute_graph<T: OrchState>(
        &self,
        graph: &ExecutionGraph<T>,
        initial_state: T,
    ) -> Result<T, LoomError> {
        let span = tracing::info_span!("loom.execute_graph", /* … */);
        let _guard = span.enter();
        let result = graph.execute(initial_state).await;
        span.record("status", if result.is_ok() { "success" } else { "failure" });
        result
    }
}

// caller
let result = LoomOrchestrator::default()
    .execute_graph(&graph, state)
    .await?;
```

**After**:

```rust
// crates/or-loom/src/infra/implementations.rs
impl<T: OrchState> ExecutionGraph<T> {
    #[tracing::instrument(name = "loom.execute_graph", skip_all, fields(status))]
    pub async fn execute(&self, initial_state: T) -> Result<T, LoomError> {
        let result = self.execute_from(&self.entry, initial_state).await;
        tracing::Span::current().record(
            "status",
            if result.is_ok() { "success" } else { "failure" },
        );
        result
    }
}

// caller
let result = graph.execute(state).await?;
```

Removes one type, one file, one indirection per crate. Users go from
"instantiate orchestrator → call method" to "call method." Status
field still recorded for `or-lens`.

#### Simplification 2 — Make `LoopTopology` actually own its binding

**Before** (current — silently broken for user topologies, #19):

```rust
// crates/or-sentinel/src/builder.rs
fn bind_topology<T, P>(builder, topology, provider, registry) -> GraphBuilder<DynState>
where T: LoopTopology, P: ConduitProvider + …
{
    let any = topology as &dyn Any;
    if any.is::<ReActTopology>()        { return bind_react(builder, provider, registry); }
    if any.is::<PlanExecuteTopology>()  { return bind_plan_execute(builder, provider, registry); }
    if let Some(r) = any.downcast_ref::<ReflectionTopology>()
                                        { return bind_reflection(builder, r, provider, registry); }
    builder // user-defined topology silently returns unbound nodes
}
```

**After**:

```rust
// crates/or-sentinel/src/topology.rs
pub trait LoopTopology {
    fn build(&self) -> GraphBuilder<DynState>;
    fn name(&self) -> &'static str;

    /// Bind handlers onto the placeholder nodes the topology declared in `build`.
    fn bind<P: ConduitProvider + Clone + Send + Sync + 'static>(
        &self,
        builder: GraphBuilder<DynState>,
        provider: P,
        registry: ForgeRegistry,
    ) -> GraphBuilder<DynState>;
}

// crates/or-sentinel/src/builder.rs
pub fn build(self) -> Result<SentinelAgent<P>, SentinelError> {
    let graph = self.topology
        .bind(self.topology.build(), self.provider.clone(), self.registry.clone())
        .build()
        .map_err(SentinelError::from)?;          // also: use `#[from]`, not `to_string()`
    Ok(SentinelAgent::from_graph(graph, self.provider, self.registry))
}
```

The `Any` trick disappears. Users who define their own topology now
receive a method they *must* implement — the compiler enforces the
contract instead of the bug being silent at runtime.

#### Simplification 3 — Make the Python `ForgeRegistry` actually invoke the Python handler

**Before** (current — handler discarded, #14):

```rust
// crates/or-bridge/src/python/forge_registry.rs
#[pyclass]
pub struct PyForgeRegistry { tools: BTreeSet<String> }

#[pymethods]
impl PyForgeRegistry {
    fn register(&mut self, name: &str, _handler: Py<PyAny>) {
        self.tools.insert(name.to_owned());     // handler dropped
    }
}
```

```python
# bindings/python/orchustr/forge.py — silently does nothing useful
registry = PyForgeRegistry()
registry.register("search", search_func)        # search_func is gone
```

**After**:

```rust
// crates/or-bridge/src/python/forge_registry.rs
#[pyclass]
pub struct PyForgeRegistry {
    handlers: HashMap<String, Py<PyAny>>,
}

#[pymethods]
impl PyForgeRegistry {
    fn register(&mut self, name: &str, handler: Py<PyAny>) {
        self.handlers.insert(name.to_owned(), handler);
    }

    fn invoke(&self, py: Python<'_>, name: &str, args: PyObject) -> PyResult<PyObject> {
        let handler = self.handlers.get(name)
            .ok_or_else(|| PyKeyError::new_err(format!("unknown tool: {name}")))?;
        handler.call1(py, (args,))
    }
}
```

```python
registry = PyForgeRegistry()
registry.register("search", search_func)
result = registry.invoke("search", {"q": "rust async"})  # actually runs search_func
```

For the Rust side calling back into Python during a sentinel run, wrap
the registry in an adapter that implements `or_forge::Tool`:

```rust
struct PyToolAdapter { name: String, handler: Py<PyAny> }

#[async_trait]
impl Tool for PyToolAdapter {
    async fn invoke(&self, args: serde_json::Value) -> Result<serde_json::Value, ForgeError> {
        Python::with_gil(|py| {
            let py_args = pythonize::pythonize(py, &args)?;
            let result = self.handler.call1(py, (py_args,))?;
            pythonize::depythonize(result.bind(py))
        }).map_err(|e| ForgeError::Invocation(e.to_string()))
    }
}
```

This collapses #14, #15, #16 into a single working pattern and lets the
Python side delete its parallel `forge.py` reimplementation.
